01 — the intention

What we're actually building, said plainly.

"Managing many agents on a flat screen is a UX problem, not a mind problem. So we give each agent a rectangle and distribute them in space — overview by where things are."

We're building Looka: a WebSpatial room where every AI agent is its own floating, draggable panel. It runs in the browser on PICO (and Vision Pro) — no app store, no headset needed to develop, and it's already spatializing live in the PICO emulator. The hard spatial parts already work — panels float, you can grab and move them, and they can hold real 3D models — so the job now is simply to give each panel an agent.

The problem

Run 20 agents in a chat feed and it collapses into noise — one column, no sense of place, nothing fits in your head.

The solution

Lift each agent into its own spatial panel and place it around the room. Spatial memory does the organizing; the overview is just there.

02 — team & ownership

Everyone owns a slice, end to end.

Each member has one feature on the frontend and one API module on the backend — same name on both sides. Read it as a gentle default, not a rule: you'll mostly work in your own folder (which quietly avoids collisions), but anyone can fix anything, anywhere, anytime. A judge still sees who did what.

"Build your card however you like — reuse the shared kit or roll your own. In the end, each panel is just HTML on the page."

Nobody guards the shared pieces either. app/ (the room shell) and shared/ (Juan's drag hook, the 3D model, any common bits) are common ground — no single owner, edited by whoever needs to. Freedom first; the structure only exists to keep us out of each other's way.

03 — architecture

Thin frontend, thinner backend, one small database.

A React SPA renders the spatial room. A small TypeScript API gives each member their own route — Frank's wraps OpenClaw; Juan's, Suvi's & Joe's wrap OpenAI. Frontend and backend share one origin, so the app just calls /api/… (Vite proxies it in dev) — no CORS to fight. SQLite (Drizzle) sits right beside the backend, and it all boots together in one Docker Compose. We keep the backend deliberately plain: clarity now, hardening later.

The first thing we wire — a fixed-prompt round-trip

One button proves the whole loop is alive before anyone builds anything fancy: click it, a fixed prompt goes to the backend, the backend hits the LLM, the answer comes back and renders in the panel. Front ↔ back ↔ model, end to end.

click “Ping agent”  →  POST /api/juan/ping (fixed prompt)  →  core → OpenAI wrapper (cheap model)  →  text  →  render in the spatial card

Why misc/reference/, not docs/? Less misleading — it holds both our own writing and the outside repos that actually help us build.

Our material (this vision, architecture, emulator guide, imported XRCC refs, the WebSpatial llms-full.txt) is committed. External repos we clone — WebSpatial samples, the OpenClaw command-center kit, SensAI kits — live under misc/reference/external/, are git-ignored (no git-in-git), and are pulled by pnpm refs:pull — a script (misc/pull-references.sh) reading misc/reference/refs.json. Everyone shares one list: find a useful reference, add a line, commit, and the team gets it on the next pull. (Submodules are the alternative; we prefer the script — easier to extend, no submodule friction.)

06 — why this shape

Best practices, bent toward a hackathon.

Grounded in current pnpm-monorepo and React structure guidance — adapted so a judge grasps it fast. The worst thing that can happen is a judge opening the repo and not getting it; naming and structure exist to prevent exactly that.

i.

Dead-clean root

Only frontend/, backend/, misc/. Deployables up front; everything else (reference, config, scripts) tucked into misc.

ii.

Ownership = a feature

Vue/feature-inspired: each member owns features/<name>, keeping their components, hooks and logic together in one place. Shared UI sits in shared/. Parallel work, zero collisions.

iii.

Progressive disclosure

A README at every level — root (the 10-s "what is this?"), then frontend/, backend/, misc/. For agents, AGENTS.md is the canonical working guide and CLAUDE.md just points to it — humans and LLMs never get lost.

iv.

Types flow one way

Backend OpenAPI → generated frontend types. No hand-maintained shared package drifting out of sync.

08 — get started

Zero clue → running → first commit.

You just cloned the repo and know nothing. Here's the shortest path to seeing it run and shipping your first change.

terminal — first run
# 1 · clone & install everything (all workspaces)
git clone <repo-url> looka && cd looka
pnpm install

# 2 · (optional) pull the shared reference repos
pnpm refs:pull

# 3 · run it all — frontend :5173 + backend :3000, together
pnpm dev

See it in space: with the PICO emulator open, run adb reverse tcp:5173 tcp:5173, browse to http://10.0.2.2:5173, and click “Run as a standalone app.” Full walkthrough in misc/reference/emulator.md.

terminal — your first change
# 4 · make your first edit — open YOUR feature folder
frontend/src/features/<you>/…     # save → hot-reloads live in the headset

# 5 · lint + format (one command, no prettier)
pnpm lint --fix

# 6 · your first commit
git switch -c <you>/first-change
git commit -am "feat(<you>): my first change"
git push

Changed the DB schema? Run pnpm db:generate — Drizzle writes the migration file; commit it, and it auto-applies on the next boot / deploy. No manual SQL.

09 — deploying (Coolify)

Push to Git, Coolify does the rest.

We deploy on our self-hosted Coolify (coolify.boxgeist.com, v4). The public repo is added once as a Docker Compose resource; every push redeploys. Front + back + SQLite are one unit on one subdomain with automatic HTTPS.

terminal — run the exact stack locally first
# prove it runs before Coolify — the same compose Coolify will build
cp .env.example .env        # OPENAI_API_KEY · OPENCLAW_TOKEN · PORT …
pnpm build                  # frontend static + backend dist
docker compose up --build   # front + back + sqlite volume
  1. Add the resource (once)Coolify → project → + New → Docker Compose; pick the public GitHub repo, file docker-compose.yml, branch main.
  2. Domain + envSet the FQDN (e.g. looka.boxgeist.com) — Traefik gives auto-HTTPS. Add OPENAI_API_KEY & OPENCLAW_TOKEN in the Environment Variables tab — never in the repo.
  3. Persist the databaseAdd Persistent Storage for backend/data so looka.sqlite survives redeploys. On each boot Drizzle's migrate() applies any new migration files — schema changes ship safely with a push.
  4. Deploy + auto-deployHit Deploy, then enable the GitHub webhook so each git push main rebuilds automatically.
